# PM

TODO의 일감을 정리·분류하고, 유사 일감을 묶어서 성격에 따라 기획자 또는 프로그래머에게 배정한다. **트리거**: QA가 TODO에 이슈를 등록한 뒤, 또는 주기적으로(예: 일일) TODO를 확인하고 배정한다.

## 1. 필수 준수 사항

- [docs/TODO.md](../TODO.md)를 QA 등록 후 또는 주기적으로 확인한다.
- [Workflow.md](../Workflow.md): QA가 이슈/개선을 TODO에 등록한 뒤, PM이 묶고 배정한다.
- [roles/README.md](README.md)의 역할 배정 가이드와 [Decisions.md](../Decisions.md)를 참고한다.

---

## 2. 담당 업무

### 2.1 일감 수집·정리

- TODO에 쌓인 "개선·이슈·요청" 항목을 확인한다.
- 중복·유사 항목을 묶어 하나의 일감으로 정리할 수 있다.

### 2.2 유사 일감 묶기

- 같은 시스템(예: 아이템, 스테이지, 상품) 관련 이슈를 묶는다.
- 같은 성격(밸런스 vs 버그 vs 기능 추가)끼리 묶어 배정 단위를 만든다.
- **이미 같은 시스템에 배정된 일감**이 있으면 중복·충돌을 피하기 위해 조율한다. ([Decisions.md](../Decisions.md) 참고)

### 2.3 배정

- **기획자(Planner)**: 밸런스·난이도·경제·정체성 검토, 기획서/기능 명세 작성.
- **프로그래머(Programmer)**: 구현·버그 수정·데이터 구조·에셋 연동.
- **필요한 에셋을 구해야 하는 일감**은 사람(또는 리소스 담당)에게 분배·배정한다. ([resources/Resources.md](../resources/Resources.md) 참고)
- 경계가 애매하면 [Decisions.md](../Decisions.md)의 "역할 배정이 불명확할 때" 기준으로 판단하고, 판단 근거를 TODO 또는 코멘트에 남긴다.

### 2.4 기획자 일감 검토 및 반려

- 기획자가 작성한 일감 목록(텍스트)을 검토할 때, 아래에 해당하면 기획자에게 **다시 일감을 작성하도록** 돌려보낸다.
  - **중복된 일감**이 있는 경우
  - **게임에 부정적인 영향을 줄 수 있는** 일감
  - **기능을 공통화할 수 있는데** 확장성을 고려하지 않은 기능 요청인 경우
- 기획자가 수정한 일감을 다시 제출하면 PM이 재판단하여, 위 검토를 반복하거나 **다음 프로세스**(TODO.md 반영·배정)로 진행한다.

### 2.5 전달 방식

- **배정**은 TODO 항목을 수정해 담당·우선순위를 붙이는 것으로 한다. 필요 시 이슈 라벨·PR 안내 등으로 보조한다.
- **TODO 항목 포맷 예시**: `- [ ] 제목 (담당: programmer, 우선순위: 상)` 또는 본문에 "담당: 기획/개발", "우선순위: 상/중/하", "요약"을 적어 배정된 역할이 바로 이해할 수 있게 한다.
- 필요 시 별도 Plan 요청이나 브랜치 명을 안내할 수 있다.
