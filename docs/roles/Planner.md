# 기획자 (Planner)

게임 밸런스·정체성 유지와 기획 문서·기능 명세를 담당한다. 기획자도 Plan을 작성하며, 기획 작업 전 Plan 리뷰어 검토를 받는다. **코드를 변경하지 않으므로 코드 리뷰 대상이 아니다** (문서·기획 변경만 해당).

---

## 1. 필수 준수 사항

- [Workflow.md](../Workflow.md) 공통 규칙 준수: 브랜치 → Plan 작성 → Plan 리뷰어 검토 → 작업 → 문서·TODO 기록.
- 작업 전 [plan/GamePlan.md](../plan/GamePlan.md)를 읽고, 변경 시 정체성·밸런스 원칙과 충돌하지 않는지 확인한다.

---

## 2. 담당 업무

### 2.1 게임 밸런스 및 정체성 유지

- 난이도 곡선, 보상, 경제(재화·상품), 성장 속도가 GamePlan의 원칙과 맞는지 검토한다.
- 새 기능·콘텐츠 제안 시 "Arrow Puzzle의 핵심 루프·타겟 경험"을 해치지 않는지 확인한다.
- 밸런스 수치·규칙 변경 시 변경 이유와 기대 효과를 문서에 남긴다.

### 2.2 기획 문서·기능 명세

- 프로그래머가 구현할 수 있도록 **기능 명세**(입력·출력·예외·UI/UX 요구)를 작성한다. **이번 작업/일감에 대응하는 명세**는 파일명(예: `FeatureSpec_Items.md`) 또는 PR/이슈 링크로 명시해 프로그래머가 참고할 수 있게 한다. 기획 명세는 프로그래머가 구현 시 참고한다.
- 기능 명세에 **필요 리소스 목록**(이름·용도·배치 경로 또는 예상 경로)을 적는다. 이미 있는 리소스는 경로를 명시한다. 상세 규칙은 [resources/Resources.md](../resources/Resources.md) 참고.
- 스테이지·아이템·상품·광고 등 콘텐츠 정의는 GamePlan 및 [Glossary.md](../Glossary.md)와 용어를 맞춘다.
- **산출물 위치**: 기획 문서·기능 명세는 `docs/plan/` 하위 또는 GamePlan 보조 문서로 둔다. 팀 정책에 맞게 통일한다.

---

## 3. 작업 후 필수

- 무엇을 바꿨는지(밸런스, 명세, 결정 사항) **[docs/changelog/](../changelog/)**에 브랜치·날짜별로 기록하거나 PR description에 적는다. ([changelog/README.md](../changelog/README.md) 참고)
- [TODO.md](../TODO.md)에 남은 작업·후속 기획·개선 아이디어를 추가한다.
