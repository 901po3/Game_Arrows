# Decisions

애매한 경우의 판단 기준, 예외 처리, 실패 시 할 동작을 기록한다. AI·팀원이 동일한 기준으로 행동하기 위함.

---

## 역할 배정이 불명확할 때

- PM이 판단; 판단 근거를 TODO 또는 PR 코멘트에 한 줄 남긴다.
- “기획 vs 개발” 경계: 데이터/규칙/밸런스 정의 → 기획자, 구현·코드·에셋 연동 → 프로그래머.

---

## Plan 리뷰 반려 시

- 반려 사유를 Plan 리뷰어가 명시한다. 작성자는 Plan 수정 후 다시 제출한다. 같은 브랜치에서 이어서 진행.

---

## 코드 리뷰 반려 시

- 반려 사유를 리뷰어가 명시한다. 작성자는 수정 후 문서(TODO·변경 요약)를 업데이트하고 다시 리뷰 요청한다.

---

## QA에서 크리티컬 버그 발견 시

- 해당 이슈를 TODO에 “긴급” 또는 우선순위 “상”으로 등록한다. PM이 즉시 배정할 수 있도록 한다.

---

## 문서 충돌(서로 다른 규칙이 보일 때)

- docs/ 내 문서가 우선. 루트나 외부 문서와 충돌하면 docs/ 버전을 기준으로 하고, 필요 시 docs/ 문서를 수정해 통일한다.

---

## 동시 작업(같은 시스템을 여러 일감이 건드릴 때)

- PM이 배정할 때 같은 시스템(아이템, 스테이지 등)에 이미 "진행 중" 또는 배정된 일감이 있으면, 순서를 정하거나 한쪽 완료 후 배정하는 등 조율한다. TODO에 진행 상태를 표시해 두면 참고하기 쉽다.

(추가 결정이 생기면 위 형식으로 섹션을 더한다.)
